# 桌面版打包、便携目录与运行时检测方案

## 1. 这份文档解决什么问题

这份文档用于统一桌面版的发布和运行策略，重点回答这些问题：

- 模型应该打包进程序，还是外挂，还是应用内下载
- `CUDA / cuDNN` 这种 GPU 依赖应该怎么处理
- 文件是否放在程序同目录
- 路径应该怎么设计
- 当前 web 版迁移到桌面版时，改造会有多大
- 桌面版应该补哪些运行时检测能力

当前项目默认技术栈：

- 前端：`React + Vite`
- 桌面壳：`Tauri`
- 本地服务：`FastAPI sidecar`
- 转写：`faster-whisper`
- 视频处理：`FFmpeg`
- 数据库：`SQLite`

---

## 2. 最终结论

## 2.1 推荐发布形态

本项目桌面版建议优先采用：

- **同目录便携版优先**
- `Tauri + Python sidecar`
- 默认 `CPU 可用`
- `GPU` 作为可选增强
- 模型、数据库、缓存、导出都放在程序目录下的相对目录中

也就是程序目录本身就是一个完整工作区，用户把整个文件夹复制到别的盘、别的电脑，理论上也可以整体迁移。

## 2.2 一句话策略

**程序本体、模型、数据库、缓存、导出统一放在程序同目录；默认 CPU 可用；GPU 自动检测；CUDA/cuDNN 不打进首发包，但要提供教程和应用内检测。**

---

## 3. 为什么这次改成“同目录便携版优先”

你提到一个很现实的点：很多用户不喜欢程序把文件散落到 `C` 盘各个系统目录里。

这个诉求非常合理，尤其对下面这些用户尤其成立：

- 喜欢绿色版软件的用户
- 会把软件放到 `D` 盘、移动硬盘、网盘同步目录的用户
- 不希望软件偷偷写 `AppData` 的用户
- 需要整体备份和整体迁移的用户

对于这个项目，“同目录便携化”其实是很适合的，因为它本身就有比较明显的本地资源概念：

- 模型
- 数据库
- 缓存
- 日志
- 导出文件
- 可选的视频库

这些内容集中放在程序旁边，用户理解成本会更低。

## 4. 程序目录应该怎么组织

## 4.1 推荐目录结构

建议桌面版最终采用下面这种结构：

```text
VideoSubtitleLearning/
  VideoSubtitleLearning.exe
  backend/
  frontend/
  ffmpeg/
    ffmpeg.exe
    ffprobe.exe
  runtime/
    python/
  models/
    faster-whisper/
      tiny/
      base/
      small/
  data/
    app.sqlite3
    settings.json
    cache/
      transcripts/
      translations/
      analysis/
    exports/
    logs/
    library/
      videos/
  temp/
```

## 4.2 各目录职责

### 程序根目录

放桌面程序主入口和相对路径的根锚点。

### `ffmpeg/`

放随程序分发的 FFmpeg 二进制。

### `runtime/`

放 Python sidecar 的运行环境或嵌入式 Python。

### `models/`

放 whisper 模型，允许用户自己添加、替换、删除。

### `data/`

放所有运行时数据：

- 数据库
- 设置
- 缓存
- 导出
- 日志
- 应用管理的视频库

### `temp/`

放临时处理文件，程序退出后可按策略清理。

## 5. 模型到底怎么放

## 5.1 三种方案

### 方案 A：模型直接打包进程序

优点：

- 开箱即用
- 首次运行不需要下载

缺点：

- 安装包体积变大
- 模型升级麻烦
- 一旦模型多了，包会迅速膨胀

### 方案 B：模型完全外挂

优点：

- 安装包更小
- 用户自由度高

缺点：

- 新手门槛高
- 容易路径错误、文件不完整

### 方案 C：程序支持模型管理

也就是：

- 可以带一个默认小模型
- 可以导入本地模型
- 可以应用内下载新模型

这是最适合长期维护的方案。

## 5.2 本项目推荐方案

推荐采用混合方案：

- 程序可以内置一个默认小模型，保证“至少可转写”
- `models/` 目录对用户开放
- 后续增加“模型管理页”

用户可以：

- 使用程序自带默认模型
- 自己把模型放进 `models/`
- 在应用内切换模型
- 删除不用的模型

## 5.3 首发阶段建议

首发不建议一口气塞很多模型。

建议：

- 最多只带一个默认小模型
- 其余模型让用户手动导入或后续下载

这样更稳。

---

## 6. GPU、CUDA、cuDNN 怎么处理

## 6.1 不建议打进首发包

首发阶段不建议把下面这些东西一起打进桌面包：

- CUDA Toolkit
- cuDNN
- 一整套 GPU 运行时依赖

原因：

- 不是每个用户都有 NVIDIA 显卡
- 与驱动、显卡、`ctranslate2`、系统环境强相关
- 容易版本不匹配
- 安装包体积会明显变大
- 维护成本高

## 6.2 推荐原则

推荐原则很简单：

- `CPU` 是保底
- `GPU` 是增强

也就是：

- 没装 `CUDA / cuDNN` 也能用
- 只是转写会慢一些
- 如果本机 GPU 环境完整，则自动启用 GPU
- 如果不完整，则自动回退 CPU

## 6.3 是否需要用户教程

需要，而且建议一定写。

这份教程至少要说明：

- 哪些用户需要配置 GPU
- 需要安装哪些版本
- 怎么验证是否安装成功
- 在应用里怎么查看当前是否真的跑在 GPU 上

## 6.4 是否还需要应用内提示

需要。

不要只给文档，还要给应用内的“运行时检测”。

这样用户不用猜。

## 7. 桌面版必须补的运行时检测

桌面版建议新增一个“运行时检测 / 环境检测”区域。

至少检测下面四类内容。

## 7.1 是否找到 FFmpeg

检测内容：

- 是否存在 `ffmpeg.exe`
- 是否存在 `ffprobe.exe`
- 是否可以正常执行 `ffmpeg -version`
- 当前使用的是哪一个路径

建议界面显示：

- `FFmpeg：已找到 / 未找到`
- `路径：...`
- `版本：...`

## 7.2 是否找到模型

检测内容：

- `models/` 目录是否存在
- 默认模型目录是否存在
- 当前设置的模型是否存在
- 模型是否可被 `faster-whisper` 正常加载

建议界面显示：

- `模型目录：已找到 / 未找到`
- `当前模型：tiny / base / ...`
- `模型状态：可加载 / 不可加载`

## 7.3 是否可用 GPU

检测内容：

- 是否检测到 NVIDIA GPU
- `nvidia-smi` 是否可执行
- 当前机器是否具备启用 GPU 的基础条件

建议界面显示：

- `GPU：已检测到 / 未检测到`
- `显卡信息：...`
- `nvidia-smi：可用 / 不可用`

## 7.4 CUDA / cuDNN 是否完整

检测内容：

- `CUDA` 运行时是否可见
- `cuDNN` 动态库是否可见
- `ctranslate2 / faster-whisper` 是否能以 `device=cuda` 真正初始化成功

这里最关键的一点是：

**不要只检测环境变量，要实际尝试初始化。**

因为真正能不能跑，最终取决于：

- 驱动
- CUDA
- cuDNN
- Python 依赖
- `ctranslate2`

是否能一起工作。

建议界面显示：

- `CUDA：可用 / 不可用`
- `cuDNN：可用 / 不可用`
- `Whisper CUDA 初始化：成功 / 失败`
- `失败原因：...`

## 7.5 建议补一个统一状态接口

后端建议新增例如：

- `GET /api/runtime/status`
- `POST /api/runtime/detect`

返回内容可包含：

- `app_root`
- `ffmpeg_found`
- `ffmpeg_path`
- `model_dir_found`
- `current_model_found`
- `gpu_detected`
- `nvidia_smi_available`
- `cuda_available`
- `cudnn_available`
- `whisper_cuda_ready`
- `effective_device`
- `effective_compute_type`
- `message`

这样前端设置页就可以直接展示。

## 8. 路径应该怎么设计

## 8.1 可以用相对路径，但不能乱用

你说“放同一目录，用相对路径寻找”这个方向是对的。

但更准确的做法不是：

- 依赖当前工作目录

而是：

- 统一以 `exe` 所在目录为根目录

也就是先求出：

- `APP_ROOT`

然后从它派生：

- `APP_ROOT/ffmpeg`
- `APP_ROOT/models`
- `APP_ROOT/data`
- `APP_ROOT/temp`

## 8.2 为什么不能依赖当前工作目录

因为桌面程序启动时，当前工作目录不一定稳定。

比如：

- 双击启动
- 从快捷方式启动
- 从命令行启动
- 被 Tauri 拉起 sidecar

这些情况下工作目录都可能不同。

所以不能写成这种假设：

```python
Path("./models")
Path("../outputs")
```

而应该写成：

```python
APP_ROOT / "models"
APP_ROOT / "data" / "cache"
```

## 8.3 对当前项目的建议

建议后端新增统一路径模块，例如：

- `backend/app/services/app_paths.py`

由它负责：

- 获取当前应用根目录
- 判断当前是开发环境还是桌面环境
- 返回所有标准目录

例如：

- `get_app_root()`
- `get_models_dir()`
- `get_data_dir()`
- `get_ffmpeg_dir()`
- `get_logs_dir()`

## 9. 当前 web 版迁移到桌面版，改造会不会很大

答案是：

- **不会特别大**
- 但会有一层基础设施改造

## 9.1 大部分业务层可以复用

当前 web 版已经比较完整，下面这些大多都可以继续复用：

- 前端主界面
- 字幕总览
- 点句解析
- 收集册
- 设置页
- 后端转写、翻译、分析 API
- 数据库存取逻辑

## 9.2 真正要改的主要是四层

### 1. 路径层

这是最重要的一层。

现在很多路径还是按仓库目录推导，桌面版要统一改成按 `APP_ROOT` 推导。

### 2. 启动层

Tauri 需要负责：

- 启动 Python sidecar
- 传递应用根目录
- 关闭时回收进程

### 3. 资源发现层

后端要知道：

- FFmpeg 在哪里
- 模型目录在哪里
- 数据库在哪里
- 输出目录在哪里

### 4. 运行时检测层

桌面版要补：

- FFmpeg 检测
- 模型检测
- GPU 检测
- CUDA/cuDNN 检测

## 9.3 前端改造不会是“推倒重来”

因为当前前端已经做了 API 根地址抽象，所以迁移成本相对较低。

也就是说：

- 不是重写 web 版
- 而是在现有 web 版外围加一层桌面运行环境适配

## 10. 推荐的发布策略

## 10.1 首发优先做便携版

既然你更希望文件不乱放，那么首发可以优先做：

- 压缩包版
- 解压即用版
- 同目录便携版

特点：

- 所有文件都在一个目录
- 用户容易理解
- 便于备份和迁移

## 10.2 便携版首发包建议包含

- Tauri 主程序
- Python sidecar
- FFmpeg
- 基础前端资源
- 一个默认小模型，或者空模型目录
- `data/` 初始目录结构

## 10.3 首发版不建议包含

- CUDA Toolkit
- cuDNN
- 多个大型模型
- 太多预置视频资源

---

## 11. 还要注意的其他问题

## 11.1 模型许可证

要确认：

- 模型是否允许再分发
- 是否需要附带许可证
- 是否适合直接打包到程序里

## 11.2 磁盘占用

这个项目会产生很多本地文件：

- 模型
- 转写缓存
- 翻译结果
- 分析缓存
- 导出文件
- 用户导入视频

所以后续建议补：

- 缓存清理
- 模型大小显示
- 导出目录管理

## 11.3 升级兼容

如果以后桌面版升级，要考虑：

- 数据库迁移
- 模型目录兼容
- 缓存目录兼容

## 11.4 代理问题

因为你已经接入了：

- DeepLX
- 各种 LLM API

所以后续桌面版也要考虑：

- 是否继承系统代理
- 是否允许单独配置代理
- 模型下载是否走代理

---

## 12. 最终推荐方案

本项目桌面版推荐采用下面这套策略。

### 目录策略

- 同目录便携版优先
- 所有运行数据放在程序目录下的相对目录中
- 统一以 `exe` 所在目录为根目录

### 模型策略

- 最多内置一个默认小模型
- 其余模型放在 `models/`
- 后续做模型管理页

### GPU 策略

- `CPU` 默认可用
- `GPU` 自动检测
- `CUDA / cuDNN` 不打进首发包
- 提供教程和应用内运行时检测

### 工程策略

- 尽量复用现有 web 版业务层
- 重点改造路径层、启动层、资源发现层、运行时检测层

---

## 13. 一句话结论

对于当前这个项目，最合适的方向不是“安装版 + 系统目录分散存储优先”，而是：

**以同目录便携版为优先目标，把程序、模型、数据库、缓存、导出统一组织在程序目录下，并通过统一路径层和运行时检测机制保证桌面版稳定运行。**
