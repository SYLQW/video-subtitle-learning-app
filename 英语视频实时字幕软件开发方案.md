# 英语视频实时字幕软件开发方案

更新日期：2026-04-06  
适用平台：Windows 10/11 桌面端  
项目定位：本地视频学习工具，支持英语语音识别、中文字幕、句子解析、词典查询与问答

## 1. 项目目标

开发一款面向英语学习者的桌面软件。用户导入本地视频后，软件能够：

- 识别视频中的英语语音
- 生成带时间轴的英文字幕
- 将字幕翻译为中文
- 以“歌词流”形式展示过去句子、当前句子和即将到来的句子
- 支持点选句子、选中文本、跳转视频时间点
- 支持词典释义、重点词汇提取、语法解释和大模型问答
- 支持后台处理整段视频并导出字幕文件

本项目首版不做画面文字 OCR，只处理视频音轨中的语音内容。

## 2. 产品形态与交互目标

### 2.1 主界面布局

按当前讨论，界面分为三块：

- 左上：视频播放区
- 右侧：实时字幕流区
- 左下：句子解析与学习区

后续可以在右下或底部留出聊天框入口，用于围绕当前句子提问。

### 2.2 核心交互

- 视频播放时，右侧字幕流按时间轴自动滚动
- 当前句高亮
- 上方显示已播放句子
- 下方显示即将播放句子
- 用户可点击任一句字幕，视频跳转到该句开始时间
- 用户可鼠标选择某句中的单词或短语
- 用户选择句子后，左下区域展示：
  - 该句中文翻译
  - 生词和释义
  - 句法/语法解释
  - 学习提示
- 用户可就当前句子向大模型继续提问

这一类“自动滚动 + 可选择文本”的实现难度是中等，不属于技术瓶颈。前端需要处理的重点是：

- 自动滚动时不要打断用户手动阅读
- 用户开始选择文本时暂停自动跟随
- 提供“跟随播放开关”，允许用户重新切回自动模式

## 4. 技术选型结论

## 4.1 桌面端架构

推荐架构：

- 前端：`React + Vite`
- 桌面壳：`Tauri`
- AI/服务层：`Python`
- 视频处理：`FFmpeg`
- 本地存储：`SQLite`

### 为什么这样选

#### React

- 适合实现复杂字幕列表、交互式学习面板、设置页和聊天区
- 生态成熟，组件和状态管理方案丰富

#### Vite

- 本地开发启动快
- 构建速度快
- 与 React、Tauri 集成顺畅

#### Tauri

- 用 Web 技术写界面，但打包成桌面应用
- Windows 上基于 WebView2，包体通常比 Electron 更轻
- 适合做“本地视频 + 本地 AI + 本地配置”的桌面软件

#### Python

- `faster-whisper`、翻译模型、文本分析工具链都更容易在 Python 生态里接
- 适合承载 AI 推理、字幕生成、问答编排等后端任务

#### FFmpeg

- 用于抽取音轨
- 用于字幕封装、转码、格式输出
- 用于生成导出文件

#### SQLite

- 保存项目配置、字幕缓存、学习记录、收藏句子等
- 单机桌面工具足够使用

## 4.2 建议的进程结构

- Tauri 主进程：桌面能力、窗口管理、系统集成
- React 前端：视频播放、字幕 UI、设置页、学习面板、聊天框
- Python 本地服务：识别、翻译、分析、导出

推荐通信方式：

- Tauri 前端 <-> Python：本地 HTTP 或 IPC
- 前端不直接保存第三方服务密钥
- 密钥统一由桌面端安全存储模块管理

## 5. 语音识别方案

## 5.1 选择 `faster-whisper`

推荐使用 `faster-whisper` 作为核心识别引擎，原因：

- 基于 Whisper 能力，识别质量成熟
- 推理速度更快
- 支持句级时间戳和词级时间戳
- 支持批处理模式
- 支持 VAD 过滤
- 适合做实时字幕和后台整段处理

### 重要说明

`faster-whisper` 的内建 `translate` 能力应按 Whisper 的范式理解，主要是“翻译成英语”，不应将其视为“任意语言语音直接翻译成中文”的完整解决方案。  
因此本项目若目标是“英语或其他语言 -> 中文字幕”，仍需要单独的文本翻译模块。

## 5.2 处理模式设计

### 实时模式

- 视频播放时分段取音频
- 识别后立即生成英文句子
- 送入翻译模块生成中文
- 推送到字幕流 UI

适合边看边学。

### 后台模式

- 整段提取音频
- 一次性或批量识别
- 输出完整时间轴字幕
- 批量翻译
- 生成完整 `srt/vtt/lrc`

适合提前处理整部视频，之后流畅学习。

### 推荐策略

首版同时支持两种模式：

- “立刻观看”：优先体验
- “后台生成完整字幕”：优先质量和稳定性

## 6. 翻译方案

## 6.1 云翻译方案

云翻译的优势：

- 接入快
- 质量稳定
- 多语言支持成熟
- 首版开发成本低

云翻译的不足：

- 需要联网
- 有字符计费成本

### 建议设计

翻译服务做成可插拔 Provider：

- `DeepLProvider`
- `GoogleTranslateProvider`
- `AzureTranslatorProvider`
- `LocalM2M100Provider`（后续）

用户在设置页中自行填写：

- 服务商
- API Key
- Endpoint
- Region（如 Azure 需要）
- 源语言
- 目标语言
- 是否自动检测语言

### 成本结论

按 2026-04-06 查到的官方公开信息：

- Google Cloud Translation：每月前 `500,000` 字符免费，超出后按量收费，公开价格页显示 `20 美元 / 100 万字符`
- Azure Translator：官方价格页显示每月 `200 万字符免费`，更细的按量价格需在定价器中按区域查看
- DeepL API Free：每月 `500,000` 字符免费；Pro 为月费加按字符计费

按工程经验推算，一小时英语视频常见大约在 `4 万到 7 万字符`，因此在免费额度内试做 MVP 压力并不大。  
上面的“一小时字符数”是经验估算，不是官方计费样例。

## 6.2 本地翻译方案

候选模型：

- `NLLB-200`
- `M2M100`

### NLLB-200

优点：

- 多语言覆盖强
- 翻译质量通常较好

注意：

- Hugging Face 模型卡显示其许可证为 `CC-BY-NC-4.0`
- 对商业落地不够友好

### M2M100

优点：

- 多语言翻译能力成熟
- Hugging Face 模型卡显示 `MIT` 许可证，更适合产品化

缺点：

- 本地推理延迟和显存占用会高于云翻译

### 首版建议

若以尽快做出可用产品为目标：

- 首版：`faster-whisper + 云翻译`
- 后续增强：增加 `M2M100_418M`

### 你这台机器上的性能判断

根据本机检测结果：

- GPU：`NVIDIA GeForce GTX 1650 Ti`
- 显存：`4 GB`
- 驱动报告的 CUDA Runtime：`12.2`

这意味着：

- `faster-whisper` 可优先尝试 `small` 或 `medium` 级别模型
- 本地翻译模型应优先考虑较轻量方案
- `M2M100_418M` 仍有尝试价值
- 更大的本地翻译模型在 4GB 显存上会更吃紧，可能需要退回 CPU 或低精度/量化方案

这一段是结合模型规模与本机硬件作出的工程推断，不是官方 benchmark。

## 8. 数据流设计

## 8.1 实时模式数据流

1. 用户打开视频
2. 视频开始播放
3. Python 服务抽取当前播放附近的音频片段
4. `faster-whisper` 识别出英文句子
5. 翻译 Provider 输出中文
6. React 前端更新字幕流
7. 用户点击句子
8. 学习区加载词典和大模型解释

## 8.2 后台模式数据流

1. 用户导入视频
2. FFmpeg 抽取整段音频
3. `faster-whisper` 整段识别
4. 按句切分
5. 批量翻译
6. 存入 SQLite
7. 导出 `srt/vtt/lrc`
8. 用户之后播放时直接读取缓存

## 10.3 Rust stable-msvc 如何安装

`stable-msvc` 的意思是：

- Rust 的稳定版工具链
- 目标平台是 Windows 的 MSVC ABI

### 安装步骤

1. 访问 Rust 官方安装页并下载安装 `rustup-init.exe`
2. 选择默认安装
3. 安装完成后重开终端
4. 运行：

```powershell
rustup default stable-x86_64-pc-windows-msvc
rustc -V
cargo -V
```

如果输出版本号，说明 Rust 安装成功。

## 10.4 如何确认或安装 Visual Studio C++ Build Tools

### 你这台机器当前情况

- 已安装 `Visual Studio Professional 2022`
- 已安装 `Microsoft Visual Studio Installer`

这说明：

- 你机器上有 Visual Studio 主程序
- 但不能仅凭当前终端就断定 C++ 工作负载是否完整安装

### 最可靠的确认方法

1. 打开 `Visual Studio Installer`
2. 找到 `Visual Studio Professional 2022`
3. 点击 `修改`
4. 查看是否勾选：
   - `使用 C++ 的桌面开发`
5. 在“单个组件”中确认至少存在：
   - `MSVC v143 - VS 2022 C++ x64/x86 build tools`
   - `Windows 10 SDK` 或 `Windows 11 SDK`
   - `C++ CMake tools for Windows`

### 命令行快速判断方法

如果你打开的是 “x64 Native Tools Command Prompt for VS 2022”，执行：

```powershell
cl
```

若能看到 Microsoft C/C++ 编译器版本信息，说明 MSVC 编译工具可用。

### 普通 PowerShell 下为什么可能找不到 `cl`

因为 `cl.exe` 常常只在：

- Developer Command Prompt
- 或加载了 Visual Studio 环境变量的终端

中可直接调用。  
所以普通 PowerShell 里 `cl.exe NOT_FOUND` 并不等价于“没有安装 C++ 工具链”。

## 10.6 FFmpeg 如何安装

### 是否必须安装

对于 `faster-whisper` 本身，官方 README 说明其底层 `PyAV` 会捆绑 FFmpeg，因此“系统级 FFmpeg 不是必需前提”。  
但对本项目来说，仍强烈建议安装系统级 `FFmpeg`，因为我们需要：

- 抽取音轨
- 导出字幕
- 视频处理与格式转换

### 安装步骤

1. 打开 FFmpeg 官方下载页
2. 按官方页面提供的 Windows builds 链接下载预编译版本
3. 解压到固定目录，例如：

```text
C:\ffmpeg
```

4. 将以下目录加入系统 `PATH`：

```text
C:\ffmpeg\bin
```

5. 重新打开终端并执行：

```powershell
ffmpeg -version
```

若能输出版本信息，则配置成功。

### 你这台机器当前情况

当前终端中：

```text
ffmpeg NOT_FOUND
```

因此需要安装或补 PATH。

## 10.7 CUDA 12 + cuDNN 9 如何配置

这一部分只在你希望用 NVIDIA GPU 加速时需要配置。

### 你这台机器当前情况

本机检测到：

- GPU：`NVIDIA GeForce GTX 1650 Ti`
- 驱动版本：`537.13`
- `nvidia-smi` 显示 CUDA Runtime Version：`12.2`
- 但 `nvcc` 当前不存在

这说明：

- 你已经有 NVIDIA 驱动
- 但尚不能确认已安装 CUDA Toolkit
- 仅有驱动不等于开发/推理所需工具链已完整

### 为什么要装 CUDA Toolkit 和 cuDNN

`faster-whisper` 当前依赖的 `ctranslate2` 在 GPU 模式下需要：

- CUDA 12
- cuDNN 9

驱动只负责让系统识别 GPU；真正给推理框架调用的开发/运行时库，还需要 CUDA Toolkit 和 cuDNN。

### 安装步骤

1. 先确认显卡驱动正常
2. 从 NVIDIA 官方下载并安装 `CUDA Toolkit 12.x`
3. 安装完成后重开终端
4. 运行：

```powershell
nvcc --version
```

若能输出版本，说明 CUDA Toolkit 已安装。

5. 再从 NVIDIA 官方下载并安装 `cuDNN 9.x`
6. 按官方安装说明完成库文件部署或安装程序安装
7. 确认 cuDNN 对应目录已进入系统可搜索路径

### 实务建议

- 优先保持 `CUDA 12.x + cuDNN 9.x` 这一对组合
- 不要随意混装更旧的 `cuDNN 8`
- 安装后最好重启一次系统

### 如何验证 GPU 推理环境

```powershell
nvidia-smi
nvcc --version
```

然后在 Python 虚拟环境中安装相关依赖并运行一次最小推理测试。

## 13. 当前推荐的首版技术决策

若以“尽快做出可用产品”为目标，推荐首版这样定：

- 桌面端：`React + Vite + Tauri`
- 识别：`faster-whisper`
- 翻译：用户自配云翻译 API
- 学习解释：用户自配大模型 API
- 导出：`srt/vtt`
- 缓存：`SQLite`

原因：

- 开发速度快
- 初版效果更稳定
- 不由软件作者承担 API 成本
- 后续仍可无缝扩展本地翻译模型
